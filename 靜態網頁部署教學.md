# 靜態網頁部署教學（Claude Code + GitHub + Cloudflare Pages + Cloudflare Domain）

## 工具分工（先搞懂每個東西是做什麼的）

| 工具 | 角色 |
|---|---|
| **Claude Code** | AI 助手，幫你在電腦上「生成程式碼」（HTML/CSS/JS），並操作 git 指令 |
| **GitHub** | 放置程式碼的雲端倉庫（repo），網頁原始檔案存在這裡 |
| **Cloudflare Pages** | 線上部署服務，自動抓取 GitHub 的程式碼，發布成一個真實網址 |
| **Cloudflare Domain** | 購買自己的網域名稱（例如 `mywebsite.com`），讓網站有自訂網址而不是預設的 `xxx.pages.dev` |

## 一、事前準備（帳號申請）

### 1. GitHub 帳號（免費）
- 網址：https://github.com/signup
- 用來存放你的網頁程式碼
- 申請時記得設定一組好記的「使用者名稱」（之後會用到）

### 2. Cloudflare 帳號（免費）
- 網址：https://dash.cloudflare.com/sign-up
- 一個帳號同時可以用 Pages（部署）跟 Domain（購買網域）
- 註冊只需要 email + 密碼，**部署網站不需要信用卡**；如果之後要買網域才需要付款方式

### 3. 安裝 Claude Code 桌面應用程式（Desktop App）
- 請依照 https://claude.com/claude-code 的說明下載並安裝桌面應用程式（不需要使用終端機）
- 安裝完成後，直接從桌面或開始選單點開「Claude Code」圖示
- 第一次開啟會請你登入 Claude 帳號（用 email 或 Google 帳號登入即可）
- 登入後，點選畫面上的「新增專案 / Open Project」按鈕，選擇你的網頁資料夾（例如桌面上的 `我的作品集` 資料夾），就可以開始在這個專案裡跟 Claude Code 對話

### 4. 安裝 Git
- 網址：https://git-scm.com/downloads
- 下載後一路「下一步」安裝即可，安裝完成後可以請 Claude Code 幫你確認：
  > 「請幫我確認電腦上有沒有安裝好 git，並告訴我有沒有需要登入 GitHub」

### 5.（選用）自己的網域名稱
- 如果想要自訂網址（例如 `mywebsite.com`）而不是用免費的 `xxx.pages.dev`，需要一個網域
- **還沒有網域的同學**：不用先做任何事，等部署完成後在步驟 5-1 直接於 Cloudflare 購買即可
- **已經有網域的同學**（在 GoDaddy、Namecheap、台灣的代理商等買過）：不用重買，部署完成後在步驟 5-2 把網域「搬家」接到 Cloudflare 即可

> **重要提醒，請務必看清楚這兩種情況的差別：**
> 1. **Cloudflare Pages 本身的部署不需要網域**：完成步驟 4 之後，你一定會拿到一個免費網址（例如 `https://你的專案.pages.dev`），這個網址可以直接打開使用，不會因為沒網域而失敗。
> 2. **但如果你的目標是要綁定「自己的網址」（例如 `mywebsite.com`）**，那麼網域就是**必要的先決條件**——沒有網域，步驟 5 的「自訂網域綁定」就一定會卡住、無法完成，因為 Cloudflare 沒有東西可以幫你綁。
>
> 所以：先確認自己這次的目標是「有免費網址就好」還是「一定要有自己的網址」。如果是後者，請先準備好網域（用 5-1 買新的，或用 5-2 搬家既有的），否則綁定自訂網域這一步會失敗。

## 二、整體流程（先理解大方向）

```
先建立 GitHub repo（備份機制就緒）
      ↓
跟 Claude Code 討論設計樣板 → AI 生成 HTML/CSS 程式碼
      ↓ 每完成一個階段就 git push 備份一次
本地網頁專案資料夾 ⇄ GitHub（持續同步備份）
      ↓ 連接
Cloudflare Pages（線上部署 → 產生 xxx.pages.dev 網址）
      ↓（選用）
Cloudflare Domain（購買自訂網域 → 綁定到 Pages）
      ↓
你的網站正式上線！
```

先建立好 GitHub repo，是為了讓接下來「設計討論、生成程式碼、反覆調整」的整個過程都有備份保護，不怕改壞或檔案遺失。之後只要請 Claude Code 修改網頁、再「推上 GitHub」，Cloudflare Pages 就會**自動**重新發布最新版本，完全不需要手動操作。

## 三、開始操作：給 Claude Code 的提示詞（照順序貼上）

打開 Claude Code 桌面應用程式，開啟你的網頁資料夾專案，在下方的對話輸入框依序貼上以下提示詞，按 Enter 送出。**每一步做完，等 Claude Code 回覆完成再做下一步。**

> 小提醒：Claude Code 在執行「建立檔案」「執行指令」等動作前，桌面應用程式通常會跳出一個確認視窗，問你是否允許這個操作（例如「允許執行 git push 嗎？」）。同學看到後**確認內容沒問題就點選「允許 / Allow」**即可繼續，這是正常的安全機制，不是出錯。

### 步驟 1：先建立 GitHub repo，把「備份」機制準備好

> **為什麼要先做這一步，而不是先設計網頁？**
> 因為接下來你會跟 Claude Code 來來回回討論設計、改程式碼，過程中檔案隨時可能因為手滑刪除、電腦當機、或不小心讓 Claude Code 改壞而遺失。**先把 GitHub 倉庫接好，之後每完成一個小階段就 push 一次**，等於是每一步都自動存了一份雲端備份，就算本機檔案出問題，也能隨時從 GitHub 拿回最近一次的版本，不用整個重做。這跟「寫報告要養成存檔習慣」是一樣的道理，只是 git/GitHub 提供的是「有歷史紀錄」的進階版存檔。

> 幾乎所有同學都是第一次用 git/GitHub，這裡先用白話解釋幾個名詞，再進入操作。不懂沒關係，照著提示詞貼給 Claude Code 就能做完，名詞解釋是讓你大概知道「現在在做什麼」。

#### 1-0　先搞懂幾個基本名詞（不用背，看過一遍有印象就好）

| 名詞 | 白話解釋 |
|---|---|
| **git** | 一個記錄「檔案修改歷史」的工具，裝在你電腦裡，跟 GitHub 不是同一個東西 |
| **repository（repo，倉庫）** | 一個專案的儲存空間，裡面放你的所有程式碼檔案 |
| **git init** | 「啟用」git，告訴電腦「這個資料夾開始要被 git 記錄歷史了」 |
| **git add** | 把改好的檔案「放進待送出的籃子裡」（準備要存檔） |
| **git commit** | 正式「存檔」，把籃子裡的檔案存成一個有紀錄的版本，並寫一句話描述這次改了什麼 |
| **git push** | 把本機存好的版本「上傳」到 GitHub 雲端，**也就是「備份」這個動作** |
| **git pull** | 把 GitHub 雲端上的版本「下載」回本機（多人協作或換電腦時才需要） |
| **.gitignore** | 一份清單，列出哪些檔案不需要被 git 記錄（例如系統產生的暫存檔） |

> 簡單流程就是：**改檔案 → add（放進籃子）→ commit（存檔並寫說明）→ push（上傳備份到 GitHub）**。從現在開始，每完成一個階段（例如：生成第一版程式碼、調整完版面、確認預覽沒問題）都重複這四步，Claude Code 都會幫你執行指令，你只需要確認就好。

#### 1-1　請 Claude Code 初始化 git 並建立 GitHub repo（這時候資料夾還是空的也沒關係）

👉 **請複製以下整段內容，貼到 Claude Code 對話框：**

```
我準備要開始做一個靜態網頁專案，我是第一次用 git 和 GitHub，
在開始寫任何網頁內容之前，請先幫我把「備份機制」設定好：
1. 在這個資料夾初始化 git（git init），並跟我說這一步做了什麼
2. 建立 .gitignore（排除不需要上傳的檔案）
3. 先建立一個簡單的 README.md，寫上專案名稱「我的個人作品集」，並幫我 git add、commit
   （commit 訊息請寫「初始化專案」），讓專案從一開始就有版本紀錄
4. 接著請告訴我，要怎麼在 github.com 網站上手動建立一個新的 repository
   （因為我需要自己登入 GitHub 操作，不能幫我自動建立帳號或 repo）
5. 等我建立好 repo 並把網址貼給你之後，幫我把本地的程式碼推上去（git push）
6. 每個步驟執行前，請先用一句話跟我說「接下來要做什麼、為什麼要做」
7. 之後請提醒我：每完成一個重要階段（例如生成程式碼、調整設計）都要記得 push 備份

我的 GitHub 使用者名稱是：[請填寫你的 GitHub 帳號名稱]
```

> Claude Code 會引導你到 https://github.com/new 手動建立一個新的 repository（記得選 Public，名稱可以跟你的專案一樣，例如 `my-portfolio`），建立好之後把網址（例如 `https://github.com/你的帳號/專案名稱`）貼給 Claude Code，它就會幫你推上去。

> **登入驗證提醒**：GitHub 現在不能直接用密碼 push，第一次 push 時通常會跳出瀏覽器要你登入授權，或是需要設定「Personal Access Token」。如果卡住，直接把畫面上的訊息複製貼給 Claude Code，請它教你怎麼設定即可（也可參考下方常見問題表）。

> 完成這一步之後，你的 GitHub repo 已經是「待命中」的備份倉庫，接下來不管網頁怎麼改、改幾次，都只要重複 add → commit → push，每次都能安心備份。

### 步驟 2：用 AI 生成網頁程式碼（以「我的個人作品集」為範例）

這個步驟分成「先設計樣板」和「再生成程式碼」兩個提示詞，不要跳過第一個，讓 Claude Code 先把設計想清楚，畫面才會好看。**每個小階段做完，記得回頭 push 備份一次（提示詞見 2-3）。**

#### 2-1　先請 Claude Code 提案設計樣板（還不要寫程式碼）

👉 **請複製以下整段內容，貼到 Claude Code 對話框：**

```
我想做一個靜態網頁，主題是「我的個人作品集」。
在開始寫程式碼之前，請先幫我設計一套網站樣板提案，先不要生成程式碼，內容包含：

1. 參考風格：請列出 3-5 位風格不同的藝術家/設計師個人網站作為參考方向
   （例如：極簡黑白風格、大膽撞色風格、手寫塗鴉風格、攝影作品集常見的全螢幕圖片風格、
   日系留白風格等），並簡單說明每種風格適合什麼樣的作品集內容。

2. 色票（Color Palette）：針對你建議的風格，提出一組具體的配色方案，
   包含主色、輔色、背景色、文字色的色碼（hex code），並說明為什麼這樣搭配。
   **請不要只給我色碼文字，務必同時用色塊（色票卡片）的視覺方式展示出來，
   讓我可以直接看到顏色長什麼樣子，不用自己腦補。**
   如果有多組方案，請每組都各自展示一份色票卡片，方便我比較。

3. 版面格式（Layout）：說明網站的區塊規劃，例如：
   - 首頁 Hero 區塊（大標題/自我介紹）
   - 作品展示區（Grid 排版 or 滑動式 or 單欄大圖）
   - 關於我區塊
   - 聯絡方式/社群連結區塊
   並說明在手機上要怎麼排列。

4. 字體建議：標題字體與內文字體的搭配建議（可使用 Google Fonts）。

請用條列方式整理成一份簡單的設計提案給我看，色票部分請務必附上實際顏色色塊（不要只寫色碼），
我確認後你再開始寫程式碼。
我的作品類型是：[請填寫，例如：插畫 / 攝影 / UI設計 / 平面設計 / 手作]
我喜歡的氛圍是：[請填寫，例如：簡約有質感 / 活潑繽紛 / 復古文青 / 科技未來感]
```

> Claude Code 回覆設計提案後，你可以直接說「我喜歡方案 2，但顏色想再深一點」之類的話來回調整，確認滿意後再進入下一步。這個討論階段不會產生程式碼檔案，不用急著 push。
>
> **小技巧**：如果第一次回覆只給了色碼文字、沒有顯示顏色色塊，可以直接補一句：「請幫我用色塊／色票卡片的方式呈現這些顏色，我想直接用看的比較顏色，不想自己想像色碼」，Claude Code 通常就會用圖卡或網頁小工具的方式把顏色實際畫出來給你看（如下圖示意）。

#### 2-2　確認樣板後，正式生成程式碼

👉 **請複製以下整段內容，貼到 Claude Code 對話框：**

```
這個設計方向我確認了，請依照我們剛剛討論的色票、版面格式、字體，
幫我在目前資料夾建立完整的 HTML 靜態網頁專案：
- index.html、style.css（如果需要互動效果可以加 script.js）
- 至少包含：Hero 區塊、作品展示區（先用 4-6 個示意作品方塊/圖片佔位）、關於我、聯絡方式
- 務必符合我們討論的色票與字體
- RWD 響應式設計，手機跟電腦瀏覽都要好看
完成後跟我說可以怎麼在瀏覽器打開預覽。
```

#### 2-3　生成完第一版程式碼，先 push 備份一次

👉 **請複製以下整段內容，貼到 Claude Code 對話框：**

```
這一版程式碼我先確認過沒問題，請幫我 git add、commit（訊息寫「完成第一版網頁」）、push 到 GitHub 備份。
```

> 養成習慣：**每次 Claude Code 幫你生成或修改完一批檔案，都先問自己「這個版本值得留下來嗎？」，是的話就請它 push 一次。** 之後不管你怎麼大幅修改，都能放心嘗試，因為隨時可以回到上一個備份版本。

### 步驟 3：本地預覽、反覆調整（調整告一段落記得 push）

👉 **請複製以下整段內容，貼到 Claude Code 對話框：**

```
請幫我在瀏覽器打開剛剛做好的網頁，讓我先看看效果。
如果我說要修改，請直接幫我改 HTML/CSS 並再打開一次給我看。
```

> 你可以一直跟 Claude Code 說「我想把標題改成...」「顏色換成...」「再加一個聯絡表單區塊」，反覆調整到滿意為止。調整到一個段落、確認效果不錯之後，記得貼上步驟 2-3 的備份提示詞（把 commit 訊息改成符合這次修改的描述），再繼續下一輪調整。

### 步驟 4：用 Cloudflare Pages 線上部署

這一步需要你自己在瀏覽器登入 Cloudflare 操作（Claude Code 沒辦法幫你登入別人的帳號），但可以請它在旁邊一步步教你：

👉 **請複製以下整段內容，貼到 Claude Code 對話框：**

```
請一步一步教我，怎麼在 Cloudflare Pages 網站上，
把剛剛我推上去的這個 GitHub repository 連接起來，
讓 Cloudflare Pages 自動部署成一個網站。
請用條列式、簡單明確的步驟告訴我要點哪裡、選什麼選項，
因為我是完全不懂技術的人。
```

操作大致流程（Claude Code 會依實際畫面引導你，以下為參考）：
1. 登入 https://dash.cloudflare.com
2. 左側選單找到「Workers & Pages」
3. 點「建立應用程式」→「Pages」→「連接到 Git」
4. 授權並選擇你剛剛建立的 GitHub repository
5. Build 設定：靜態 HTML 網頁通常**框架預設選 None**、Build command 留空、輸出目錄填 `/`（或你的網頁檔案所在資料夾）
6. 點「儲存並部署」，等待 1-2 分鐘
7. 完成後會拿到一個免費網址，例如 `https://你的專案.pages.dev`

### 步驟 5（選用）：擁有並綁定自己的網域

如果你不想用 `xxx.pages.dev`，想要像 `mywebsite.com` 這種自訂網址，**前提是要先有一個網域**。這裡分兩種狀況，先確認你屬於哪一種：

- **狀況 A：我還沒有任何網域** → 走 5-1，直接在 Cloudflare 購買一個新的
- **狀況 B：我已經在其他平台（GoDaddy、Namecheap、台灣的 PChome/GoDaddy/中華電信等）買過網域了** → 走 5-2，把網域「搬家」（移轉 nameserver）到 Cloudflare 管理，不需要重買

#### 5-1　狀況 A：在 Cloudflare 購買全新網域

👉 **請複製以下整段內容，貼到 Claude Code 對話框：**

```
我想幫這個已經部署在 Cloudflare Pages 上的網站，
購買一個自己的網域名稱並綁定上去（我目前還沒有任何網域）。
請一步一步教我：
1. 怎麼在 Cloudflare 的 Domain Registration 搜尋並購買網域
2. 買好之後怎麼把這個網域連接到我的 Pages 專案
請用條列式、簡單明確的步驟說明，因為我是完全不懂技術的人。
```

操作大致流程（參考，實際畫面以 Cloudflare 為準）：
1. 登入 https://dash.cloudflare.com，左側選單找「Domain Registration」
2. 搜尋你想要的網域名稱（例如 `mywebsite.com`），確認可購買
3. 結帳購買（需要信用卡或 PayPal）
4. 購買完成後，回到你的 Pages 專案 →「自訂網域」（Custom domains）
5. 輸入剛買的網域，Cloudflare 會自動幫你設定好 DNS（因為網域跟 Pages 都在同一個 Cloudflare 帳號，通常一鍵就能綁定）
6. 等待幾分鐘到數小時讓 DNS 生效，之後就能用自己的網域開啟網站

#### 5-2　狀況 B：把已經買好的網域「搬家」到 Cloudflare（移轉 Name Server）

> 這個方法**不需要重買網域**，只是把網域的「DNS 管理權」轉交給 Cloudflare，原本的網域所有權還是你的，到期還是跟原本的平台續費（除非你之後選擇連同所有權一起轉移過去，那是另一個進階選項，不影響這裡的操作）。

👉 **請複製以下整段內容，貼到 Claude Code 對話框：**

```
我已經在其他平台（例如：[請填寫，如 GoDaddy / Namecheap / PChome / 中華電信]）
擁有一個網域：[請填寫你的網域名稱，例如 mywebsite.com]，
現在想把它接到我部署在 Cloudflare Pages 上的網站。
請一步一步教我：
1. 怎麼先在 Cloudflare 把這個網域「加入」(Add a site)
2. Cloudflare 會給我一組新的 Name Server（NS）位址，要怎麼回到原本買網域的平台去修改設定
3. 修改完之後，怎麼確認 Name Server 已經生效
4. 生效後，怎麼把這個網域綁定到我的 Pages 專案
請用條列式、簡單明確的步驟說明，並提醒我哪些步驟是要回到原本的購買平台操作。
```

操作大致流程（參考，實際畫面依各平台而異）：
1. 登入 https://dash.cloudflare.com，點「Add a site」，輸入你已經擁有的網域名稱
2. Cloudflare 會自動掃描現有的 DNS 紀錄，確認沒問題後繼續
3. Cloudflare 會提供 **兩組 Name Server**（例如 `xxx.ns.cloudflare.com`），把它們記下來
4. 回到你原本「購買網域」的那個平台（不是 Cloudflare），登入後找到該網域的「Name Server / DNS 設定」，
   把原本的 Name Server **整組換成** Cloudflare 提供的那兩組
5. 儲存後等待生效，通常幾分鐘到 24-48 小時不等（可以在 Cloudflare 後台看到狀態從「待處理」變成「使用中」）
6. Name Server 生效後，回到 Cloudflare 的 Pages 專案 →「自訂網域」（Custom domains），輸入這個網域並綁定
7. 等待 DNS 設定完成，之後就能用這個網域開啟你的網站

> **注意**：步驟 4 一定要回到「原本購買網域的那個平台」去改 Name Server，Cloudflare 沒辦法幫你登入別的平台操作，這部分 Claude Code 只能用文字教你怎麼找到設定頁面。如果找不到入口，把該平台的網址跟畫面狀況描述給 Claude Code，請它協助你找路徑。

👉 **請複製以下整段內容，貼到 Claude Code 對話框：**

```

推上去之後，Cloudflare Pages 會自動偵測並重新部署，**不需要再手動做任何事**，等 1-2 分鐘重新整理網頁即可看到最新版本（自訂網域也會同步更新）。

### 步驟 7（進階、選用）：SEO、Sitemap、Google Search Console、PageSpeed 測試

> 這一步建議**等網站確定上線、內容穩定後再做**，不用在第一次部署時就考慮。目的是讓 Google 找得到你的網站、搜尋結果顯示更完整，並檢查網站速度。

#### 7-1　什麼是 SEO？

**SEO（Search Engine Optimization，搜尋引擎最佳化）** 指的是一系列讓網站更容易被 Google 等搜尋引擎理解、收錄，並在搜尋結果中獲得較好排名的做法。簡單說：**SEO 做得好，別人搜尋相關關鍵字時比較容易找到你的網站。**

**為什麼對個人作品集網站重要：**
- 別人想找你（例如搜尋「你的名字 + 攝影師」「你的名字 作品集」）時，網站能不能出現、排在前面，直接影響能不能被看見
- 分享到社群平台（FB、LINE、Threads）時，有沒有漂亮的預覽卡片（標題、描述、縮圖），會影響別人願不願意點開

**SEO 大致分兩塊：**

| 類型 | 說明 | 對應本教學的哪些步驟 |
|---|---|---|
| **站內 SEO（On-page）** | 網站本身的設定與內容品質：標題、描述、關鍵字、圖片是否有替代文字、網站結構是否清楚、有沒有 sitemap 讓搜尋引擎好讀取 | 7-2（基本 SEO 設定）、7-8（PageSpeed 速度優化） |
| **站外 SEO（Off-page）** | 網站之外的因素：有多少其他網站／社群連結指向你的網站（稱為「反向連結」），會影響搜尋引擎對你網站的信任度 | 不在本教學範圍，例如把網址放在 Instagram bio、Behance 個人頁等 |

**常見的迷思：SEO 不是「做了就保證排名第一」**
- SEO 是長期累積的過程，沒有任何方法能保證一定排第一名
- 真正影響排名的因素很多（內容品質、關鍵字相關度、網站速度、外部連結數量、使用者停留時間等），SEO 設定是「打好基礎」，不是「捷徑」
- 後面 7-6、7-7 的 Google Search Console、7-8 的 PageSpeed Insights，都是圍繞著「把站內 SEO 基礎打好」這個目標展開的工具

#### 7-2　請 Claude Code 幫網站加上基本 SEO 設定

```
我的個人作品集網站已經上線了，網址是：[請填寫，例如 https://你的專案.pages.dev 或自訂網域]
請幫我檢查並補上基本的 SEO 設定，包含：
1. 在 index.html 的 <head> 加上完整的 title、meta description
2. 加上 Open Graph 標籤（og:title, og:description, og:image, og:url），
   讓網站分享到社群（如 Facebook、LINE、Threads）時會顯示漂亮的預覽卡片
3. 加上 favicon（網站小圖示）的設定方式說明
4. 幫我產生一份 sitemap.xml（網站地圖），列出網站所有頁面

```

#### 7-3　Sitemap（網站地圖）的功用是什麼？

剛剛 7-2 提示詞請 Claude Code 產生的 `sitemap.xml`，其實是給「搜尋引擎」看的，不是給人看的，先了解它在做什麼，後面提交給 Google 時才知道意義。

**Sitemap 是什麼：**
一份用 XML 格式寫成的清單，列出網站裡所有「希望被搜尋引擎收錄」的頁面網址，可以想成是給 Google 的「網站目錄／地圖」。

**主要功用：**

| 功用 | 說明 |
|---|---|
| 幫搜尋引擎快速找到所有頁面 | 不用讓 Google 自己一個連結一個連結慢慢爬，直接給它一份完整清單，尤其網站頁面多或彼此連結較少時特別有用 |
| 加速新頁面被收錄 | 新增頁面後，更新 sitemap 並重新提交，能讓 Google 更快發現並收錄新內容，不用乾等自然爬取 |
| 提供頁面的額外資訊 | sitemap 可以附加每個頁面「最後更新時間」等資訊，幫助搜尋引擎判斷哪些內容是新的、該優先重新檢查 |
| 避免重要頁面被遺漏 | 對於只有單頁（one-page）的個人作品集網站來說效果有限，但如果之後擴充成多頁網站（例如獨立的作品詳情頁），sitemap 的重要性會明顯提升 |

**小提醒：**
- 對於現在這種**只有一頁**的個人作品集網站，sitemap 的效益相對沒有那麼大（因為頁面本來就只有一個，Google 通常很快就能找到），但養成習慣準備好它，之後網站擴充成多頁時就不用臨時補做。
- sitemap 本身**不會影響排名高低**，它只負責「讓 Google 知道有哪些頁面存在」，跟 7-1 提到的 SEO 排名因素是兩件事。
- sitemap.xml 產生好之後，會在 7-7 提交到 Google Search Console。

#### 7-4　robots.txt 的功用是什麼？它對圖片的著作權有影響嗎？

7-2 提示詞裡也請 Claude Code 產生了一份 `robots.txt`，這份檔案跟 sitemap 一樣是給「搜尋引擎的爬蟲程式」看的，但作用方向相反，先了解清楚再往下做。

**robots.txt 是什麼：**
放在網站根目錄（例如 `https://你的網址/robots.txt`）的一份純文字規則檔，用來告訴搜尋引擎的爬蟲程式「哪些頁面/資料夾可以爬、哪些不要爬」。它是一個**禮貌性的約定**，正派的搜尋引擎（Google、Bing 等）會遵守，但不是強制的技術鎖。

**主要功用：**

| 功用 | 說明 |
|---|---|
| 允許/禁止特定頁面被爬取 | 例如不想讓後台管理頁、測試頁、草稿頁出現在 Google 搜尋結果中，可以在 robots.txt 裡指定排除 |
| 節省爬蟲資源 | 避免搜尋引擎浪費時間爬取不重要的頁面（如重複內容、系統檔案），讓它優先檢索真正重要的內容 |
| 指向 sitemap 位置 | 通常會在 robots.txt 裡寫一行指向 `sitemap.xml` 的位置，方便爬蟲找到完整頁面清單 |

對於本教學的個人作品集網站，因為只有一頁、沒有需要隱藏的後台頁面，robots.txt 內容會很簡單，主要就是「允許全部頁面被索引」+「指向 sitemap」。

**重要觀念：robots.txt 對圖片著作權「完全沒有保護作用」**

這是常見的誤解，務必弄清楚：

- ❌ **robots.txt 不是著作權保護機制**。就算你在 robots.txt 裡寫「禁止爬取圖片資料夾」，頂多只是讓 Google 圖片搜尋等正規搜尋引擎**不會主動收錄索引**你的圖片，但：
  - 任何人只要打開你的網頁，瀏覽器本來就會把圖片下載到他的電腦上（這是網頁運作的基本原理），robots.txt 完全擋不住這件事
  - 惡意的人或工具本來就可以選擇無視 robots.txt 規則（它只是君子協定，沒有強制力），直接抓取圖片
  - 別人還是可以用「另存圖片」「螢幕截圖」等方式取走你的作品
- ✅ **作品的著作權保護，從你創作完成那一刻就自動存在**（多數國家含台灣採創作完成主義，不需要登記），跟有沒有設定 robots.txt 無關。robots.txt 只是「搜尋引擎索引行為的引導」，不是版權鎖。

**如果真的想降低圖片被未經授權使用的風險，較有效的做法是（不在本教學程式碼範圍內，可另外請 Claude Code 協助）：**
- 在作品圖片上加浮水印（watermark）
- 上傳到網站的圖片解析度刻意調低、不提供原始高解析度檔案
- 在網站明顯處加上版權聲明文字（例如「© 2026 你的名字，圖片版權所有，禁止未經授權使用」）
- 重要/高價值作品避免完整公開在網路上，僅提供縮圖或部分作品

> 簡單比喻：robots.txt 像是在門口貼一張「請勿進入」的告示牌——願意遵守規則的人（正規搜尋引擎）會配合，但它不是鎖，擋不住真的想進來的人。要保護作品，得靠著作權法本身、浮水印等實際技術手段，robots.txt 解決的是完全不同層面的問題（搜尋引擎索引行為）。

#### 7-5　版權聲明 / 著作權頁面該怎麼做？

7-4 提到「在網站明顯處加上版權聲明文字」是降低圖片被盜用風險的做法之一，但很多同學不知道實務上具體要寫什麼、放在哪裡。這裡分兩個層次說明：**最簡單的版權宣告**，以及**更完整的著作權頁面**。

**第一層：頁尾版權宣告（建議每個網站都要有，最簡單）**

通常放在網頁最底部（footer），格式固定，只要照樣造句即可：

```
© 2026 你的名字. All rights reserved.
```

或中文版：

```
© 2026 你的名字　版權所有，禁止未經授權轉載
```

- `©` 符號 + 年份 + 名字 是國際通用的標準格式，**不需要登記、不需要申請，寫上去就算是正式聲明**
- 年份建議寫「網站建立年份」或「最近更新年份」，例如 2026
- 如果不確定要不要附中文，建議中英文都放（國際友善 + 在地讀者都看得懂）

👉 **如果想請 Claude Code 直接幫你加上去，可以複製以下內容貼到對話框：**

```
請幫我在網頁的頁尾（footer）加上版權宣告文字：
「© 2026 [請填寫你的名字]. All rights reserved.」
並確保每個作品的圖片上方或下方有清楚標示版權聲明（如果還沒有的話）。
加完後幫我 commit、push。
```

**第二層：獨立的「著作權聲明」頁面（進階、選用）**

如果你想更完整一點，可以另外做一個簡單的著作權說明區塊或子頁面，內容通常包含：

| 項目 | 範例內容 |
|---|---|
| 著作權歸屬 | 「本網站所有圖片、文字、設計，除特別標明外，著作權均為 [你的名字] 所有」 |
| 使用限制 | 「未經本人書面同意，禁止以任何形式重製、散布、改作本網站之圖片或文字內容」 |
| 授權聯絡方式 | 「如需轉載或商業合作使用，請透過 [Email] 與我聯繫」 |
| 例外聲明（如果有） | 例如某些作品採用創用CC授權（Creative Commons），需特別標明授權類型 |

👉 **如果想請 Claude Code 幫你做這個區塊，可以複製以下內容貼到對話框：**

```
請幫我在網站新增一個簡單的「著作權聲明」區塊（放在關於我或聯絡方式區塊附近即可，不用獨立成新頁面），
內容包含：
1. 著作權歸屬聲明（本網站圖片文字著作權均為 [請填寫你的名字] 所有）
2. 使用限制（未經同意禁止重製、散布、改作）
3. 如需合作或轉載的聯絡方式（[請填寫你的 Email]）
請用簡潔、不會太生硬的方式呈現文字，並符合我們網站的整體設計風格。
完成後幫我 commit、push。
```

**常見疑問：自己寫版權聲明，跟去申請著作權登記，效果一樣嗎？**

- 不一樣，但這不影響你現在要不要做。
- 台灣（以及多數國家）採「創作完成主義」：作品一旦完成、以具體形式呈現（例如拍出來的照片、寫好的文字），著作權就**自動產生**，不需要登記、不需要繳費、也不需要寫聲明才算數。
- 那「寫版權聲明」的實際作用是什麼？
  1. **嚇阻作用**：明確告訴瀏覽網站的人「這是有主的，不能隨便拿」，減少別人「不知道是別人作品」而誤用的情況
  2. **舉證輔助**：如果未來真的發生爭議，網站上清楚標示的日期、聲明文字，可以作為輔助證據之一（雖然不是唯一證據）
  3. **專業形象**：對作品集網站來說，有版權聲明會讓網站看起來更完整、更專業

> 簡單結論：**版權聲明不是法律必要步驟，但是低成本、高 CP 值的自我保護動作**，幾分鐘就能加上去，建議每位同學的作品集網站都做這一步。

#### 7-6　Google Search Console 是什麼？它對「搜尋自己名字」有什麼影響？

在動手提交網站之前，先搞懂這個工具實際在做什麼，避免有錯誤期待。

**Google Search Console（GSC）的主要作用：**

| 作用 | 說明 |
|---|---|
| 提交網站給 Google 收錄 | 新網站不會自動出現在 Google 搜尋結果中，透過 GSC 提交 sitemap、要求 Google 來「爬」網站，可以加速被收錄（從原本可能要幾週，縮短到幾天內） |
| 查看搜尋表現數據 | 哪些關鍵字搜尋後你的網站有出現（曝光次數）、有多少人點進來（點擊次數）、平均排名第幾。例如可以查到搜尋「你的名字 + 攝影師」時網站有沒有出現、排第幾名 |
| 發現並修正技術問題 | 行動裝置相容性、網頁載入錯誤（404）、安全性問題（如被植入惡意程式）等 |
| 管理搜尋結果顯示方式 | 設定哪些頁面要被收錄、哪些不要；確認搜尋結果顯示的標題/描述是否正確 |

**重要觀念釐清：GSC 不會直接提升排名**

- ❌ 註冊 Google Search Console、把網站加進去，**不會**讓你的網站自動排名更前面，也不代表搜尋「你的名字」就會立刻看到你的作品集。GSC 只是監控與溝通工具，不是排名演算法的一部分。
- ✅ 但它能**間接**幫助提升被搜尋到的機會：
  1. **加速被收錄**——不用 GSC，Google 可能要很久才自然發現新網站；用了之後通常幾天內就能出現在搜尋結果中。
  2. **讓你知道要優化什麼**——例如發現「攝影師 你的名字」這個關鍵字有曝光但沒人點擊，可能代表標題/描述不吸引人，可以針對性修改網頁的 `<title>` 和 meta description。
  3. **確認沒有技術障礙阻擋收錄**——例如不小心設定了禁止 Google 爬取的標籤，GSC 會提醒你，否則網站可能完全不會出現在搜尋結果中。

**真正會影響「搜尋自己名字」排名的因素：**
- 網站內容是否清楚提到你的姓名、作品關鍵字
- `<title>`、meta description 是否寫得吸引人且包含關鍵字（對應前面 7-2 加的 SEO 設定）
- 是否有其他網站連結到你（例如 Instagram bio、Behance 個人頁連回你的網站）
- 網站載入速度、是否支援手機瀏覽（對應 7-8 的 PageSpeed 測試）
- 內容的更新頻率與原創性

> 簡單比喻：GSC 像是「健檢報告 + 信箱」，告訴你 Google 怎麼看你的網站、哪裡需要改善，但它本身不是讓你「變健康」的藥——真正影響排名的還是網站內容品質、關鍵字使用、外部連結等因素。

#### 7-7　把網站提交到 Google Search Console

這一步需要你自己登入 Google 操作（Claude Code 沒辦法登入你的 Google 帳號），請它在旁邊教你：

👉 **請複製以下整段內容，貼到 Claude Code 對話框：**

```
請一步一步教我，怎麼把我的網站提交到 Google Search Console，
讓 Google 開始收錄這個網站，並送出剛剛產生的 sitemap.xml。
請用條列式、簡單明確的步驟說明，因為我是完全不懂技術的人。
```

操作大致流程（參考，實際畫面以 Google 為準）：
1. 前往 https://search.google.com/search-console
2. 用你的 Google 帳號登入，新增資源（Property），輸入你的網站網址
3. 進行「驗證」：通常會請你把一段驗證碼貼到 index.html 的 `<head>` 裡
   （可以直接請 Claude Code：「請幫我把這段 Google 驗證碼加到 index.html 裡，然後幫我 commit、push」）
4. 驗證成功後，在左側選單找到「Sitemap」，貼上 `https://你的網址/sitemap.xml` 並送出
5. 等待幾天讓 Google 開始收錄，可以在 Search Console 查看收錄狀況

#### 7-8　為什麼要使用 Google PageSpeed Insights？它對 SEO 有什麼影響？

**為什麼要測試網站速度：**

| 原因 | 說明 |
|---|---|
| 使用者體驗 | 網站載入超過 3 秒，多數訪客會直接離開，尤其手機網路環境下更明顯。對作品集網站來說，慢速載入會讓人在看到作品之前就先關掉頁面 |
| 找出具體問題點 | PageSpeed Insights 會列出具體拖慢速度的原因（圖片太大、CSS/JS 沒壓縮、字體載入阻塞等），不是只給一個籠統分數，方便針對性修改 |
| 行動裝置體驗 | 報告會分別跑「行動裝置」與「電腦」兩種分數，現在多數人用手機瀏覽網站，行動裝置分數通常更該優先優化 |

**它對 SEO 的影響：**

- Google 自 2021 年起把「網頁體驗（Core Web Vitals）」正式納入搜尋排名的評分因素之一，這些指標包括：
  - **載入速度**（多快能看到主要內容）
  - **互動反應速度**（點擊/滑動後多快有反應）
  - **畫面穩定度**（內容會不會一直跳動位移）
- PageSpeed Insights 報告中的 **Performance** 分數，正是反映這些 Core Web Vitals 指標。
- **同樣內容、同樣關鍵字優化程度的兩個網站，載入速度較快的那個，在 Google 搜尋排名上通常會比較有利**——尤其在行動裝置搜尋結果中影響更明顯。
- 報告中的 **SEO** 分數則是檢查更基本的 SEO 體質（例如有沒有 meta description、圖片有沒有 alt 文字、連結是否可被讀取等），跟 7-2 加的 SEO 設定直接相關。

> 簡單來說：PageSpeed 分數不是「裝飾用的數字」，**Performance 分數會直接影響 Google 排名**，而 SEO 分數則是檢查你的網站有沒有把基本的 SEO 設定做好。兩者都跟「能不能被搜尋到、搜尋到後排名好不好」有實際關聯，不是兩件分開的事。

👉 **請複製以下整段內容，貼到 Claude Code 對話框：**

```
請教我怎麼用 Google PageSpeed Insights 測試我的網站效能，
並說明報告裡的分數（Performance、Accessibility、Best Practices、SEO）大概代表什麼意思。
```

操作大致流程：
1. 前往 https://pagespeed.web.dev
2. 貼上你的網站網址，點「分析」
3. 會分別跑出「行動裝置」與「電腦」兩種分數報告
4. 如果分數偏低，把報告截圖或文字結果貼給 Claude Code，請它幫忙優化：

👉 **請把下面的範本貼上，並換成你自己的報告內容：**

```

請幫我看看有哪些可以優化的地方（例如圖片太大、CSS/JS 沒壓縮等），
